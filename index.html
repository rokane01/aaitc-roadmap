ARBOR AI Roadmap
AAITC, ARLO & ACE

The Arbor AI Tool Chest is where every skill lives. ARLO is the outward-facing brand that deploys those skills. Two pillars: AAITC (ops infrastructure) and ACE (AI-powered CRM communication layer). Flexible on Dynamics. Portable by design.

Sources: Ryan<>Mike 04/20, Project(s) Alignment 04/21, Patty Charter 04/16 Prepared: 04/21/26

Framing

The real question isn't "how do we build AI tools inside Dynamics." It's how we build a skills-based AI layer that's flexible enough to stay on Dynamics, extend it, or leave it — without rebuilding along the way.

This iteration surfaced that department heads, leadership, managers, and contractors were all operating from different assumptions about where tools had to live and how they'd connect. That's on me — I didn't align the full team or establish a clear vision before hitting start. The goal now is to close that gap with one shared architecture and one shared operating model everyone can build against.

Skills are the units. AAITC is where every skill lives. ARLO is the deployment surface — app or sidecar depending on user and need. Its dashboard and skills depend on the user's role. Current domain apps are The CODE and the Marketing Dashboard, and more can be designed as needed. ACE is the AI-powered communication layer that runs across every CRM-based system at Arbor — the overarching goal being maximum agentic use in personalized funnel management.

Platform Flexibility (Not a Decision — Yet)

We don't have the bandwidth for a platform shift right now. What we need is flexibility around the current system, not a forced choice.

Stay / Extend / Leave is a real question but it surfaces later — likely Q4 or beyond — when we have the capacity to take a shift on. Until then, the principle is flexibility: every skill we build must work whether Dynamics stays the backbone, gets extended, or eventually gets replaced. The architecture decision gets made in a separate project when the timing is right.

POSTURE NOW

Build flexible, defer decision

Skills portable. UI external. Dynamics via API. Keep all platform options open without committing bandwidth to a migration.

NEAR-TERM OPTION

Enhance what we have

Creative external UI work that improves EiOS / ARP / AIS / HR / LoanVision / Business Central. Most cost-effective, easiest to manage.

FUTURE

Revisit Q4+

Platform viability study kicked to its own project when bandwidth exists. No forced decision in this roadmap.

Architecture — The Skills Model

How the pieces fit

Work flows top-down: job context feeds skills, skills live in AAITC, AAITC stays alive through continuous update. Skills are then deployed to users one of two ways — through ARLO (default, role-keyed) or through a domain app (when the use case warrants its own surface, like The CODE or the Marketing Dashboard).

LAYER 01
Job Context — role-specific folders following the AI Operating Model template. ROLE / WORKFLOWS / DECISIONS / FRICTION / EXAMPLES / TOOLS / PROMPTS. Markdown-first. Living documents. Governed by Patty's input-quality standards.
LAYER 02
Skills — discrete capabilities with prerequisite dependencies. "Label & Classify Docs" before "Draft CD." Each skill pulls from job context, writes back to systems of record. Composable, versioned, role-keyed.
LAYER 03
AAITC — the Arbor AI Tool Chest. Every skill built at Arbor lives here. Central source of truth for what exists, who owns it, and how it connects to ARLO, domain apps, and external systems.
LAYER 04
Continuous Update Layer — skills don't ship once. Monthly AI Operations Review re-scores the backlog. Each tool has a steward responsible for ongoing health. Friction logs feed new skill requests. This is how AAITC stays alive instead of becoming a museum.
LAYER 05
ARLO — the outward-facing brand. Deploys skills from AAITC. App or sidecar depending on user and need. Dashboard + skill bundle keyed to the user's role. Processors see processor skills; LOAs see LOA skills; Ryan sees Ryan skills.
LAYER 06
Domain apps — current apps are The CODE (referral partner development) and the Marketing Dashboard. More can be designed as needed. Most role-specific capability lives as skills inside ARLO, not as standalone apps. ACE powers communication sequences across the stack.

The Two Product Pillars

PILLAR 01 // OPS

AAITC — Arbor AI Tool Chest

Infrastructure. Every skill lives here. Governed by the AI Operating Model. Deploys via ARLO.

WHAT IT IS

AAITC is the central inventory and source of truth — not a dashboard layered on top of existing tools, but the place every skill at Arbor actually lives. ARLO, The CODE, the Marketing Dashboard, and external tools all pull from it. This is the moat.

GOVERNED BY THE AI OPERATING MODEL

Patty + Mike co-own the framework every department builds against. Four deliverables, 4-week cadence:

  • Folder template — canonical structure every role uses (draft: ROLE / WORKFLOWS / DECISIONS / FRICTION / EXAMPLES / TOOLS / PROMPTS). Patty finalizes with input from Jason + Mike.
  • Input quality guide — how to capture context well. The "garbage in, garbage out" antidote. Patty authors.
  • Retrieval architecture — RAG vs. Claude Projects/Skills vs. MCP connectors. Patty + Mike co-own; Jason consults. Week 2 one-pager.
  • Governance — every deployed tool needs builder, owner/steward, adoption plan, measurable outcome. Continuous AI accountability, not traditional launch-and-forget. Patty + Mike co-own.

PILLAR 02 // CRM COMMUNICATION

ACE — Arbor Communication Engine

The AI-powered communication layer for every CRM-based system at Arbor. Six stages. Every audience. Maximum agentic use in personalized funnel management.

WHAT IT IS

ACE is the AI-powered communication layer for every CRM-based system at Arbor. The overarching goal is maximum agentic use in personalized CRM funnel management — every prospect, every client, every partner, every recruit gets moved through their funnel with AI-driven communication tuned to them and delivered in the voice of the Arbor person who owns the relationship.

Same six-stage flow regardless of audience (recruits, realtors, CPAs, FAs, insurance prospects, existing clients, past clients). Every Arbor system with a CRM dimension — ARP, The CODE, EiOS, AIS, and anything we add — runs through ACE for the communication piece.

USER VOICE LIBRARY

The hardest part of agentic CRM communication is making AI-driven outreach sound like the actual person on the account — not a generic brand voice. ACE solves this by building a context library for each relationship owner (LO, recruiter, insurance agent, partner rep, executive): their phrasing, their isms, their tone, their go-to analogies, how they open, how they handle objections, how they close.

Every outbound communication — comment, reply, DM, script, follow-up — is generated in the voice of the relationship owner the engagement is tied to. The prospect or client never experiences a brand voice; they experience Ryan's voice, or Jess's voice, or David's voice. Agentic conversation flows stay in character through the entire sequence.

/voice-library/[owner-name]/
├── TRAITS.md — tone, pace, personality markers
├── ISMS.md — signature phrases, metaphors, vocabulary
├── OPENS/ — how they start conversations
├── OBJECTIONS/ — how they handle pushback
├── CLOSES/ — how they ask for the next step
├── TRANSCRIPTS/ — Grain + email source material
└── EXAMPLES/ — exemplar outreach they've approved

Source material: Grain meeting transcripts, sent email, LinkedIn posts, approved scripts. Same job-capture protocol as every other AAITC skill. The owner reviews and approves exemplars; the library stays current through the 30-day iteration cycle.

Where it lives: Voice profiles are skills in AAITC. ACE consumes them at Stage 03 (Segment & Match) to assign the right voice, then Stages 04–06 generate all output in that voice.

Strategic Principle

Build skills, keep them portable, and use creative external UI to enhance what we already have.

Until we have the bandwidth for a platform shift, the discipline is:

→ Skills live in AAITC, not duplicated inside domain apps
→ The CODE and the Marketing Dashboard are our current domain apps. More can be designed as needed; most capability lives as skills inside ARLO keyed to role
→ UI lives outside Dynamics as a default
→ Creative external UI to enhance existing systems (EiOS, ARP, AIS, HR, LoanVision, Business Central) is explicitly encouraged — often the most cost-effective move
→ Data model is ours, accessed via API to whatever's the backbone today
→ Markdown-first storage (SharePoint is today's location, not tomorrow's coupling)

Domain apps can standalone or connect — both are fine. The test is whether the skill underneath survives a platform change. If yes, ship it. If no, refactor before shipping.
